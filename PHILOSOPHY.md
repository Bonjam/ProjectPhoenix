# Project Phoenix Philosophy

## Mission

Project Phoenix exists to make Docker disaster recovery simple, lightweight, and reliable.

It is designed for self-hosters who want confidence that they can recover quickly after hardware failure or data loss.

---

# Our Priorities

Every decision made in this project should follow these principles.

## 1. Recovery First

Project Phoenix is not simply a backup tool.

Its purpose is successful recovery.

If a feature improves recovery confidence, it belongs.

If it does not, it probably does not belong.

---

## 2. Lightweight

Project Phoenix should run comfortably on:

- Raspberry Pi
- Mini PCs
- Home servers
- NAS devices

Avoid unnecessary dependencies.

Prefer Bash and standard Linux tools.

---

## 3. Human Readable

Configuration should be readable.

Reports should be readable.

Logs should be readable.

No databases are required for the core application.

---

## 4. Modular

Each module should do one job well.

Modules should remain independent wherever possible.

---

## 5. Privacy

Project Phoenix never requires:

- Cloud accounts
- Telemetry
- Analytics
- User tracking

The user's data belongs to the user.

---

## 6. Open

The project welcomes ideas, improvements, testing, and contributions.

Code should be understandable before it is clever.

---

## 7. Stability Before Features

A reliable backup is more valuable than ten unfinished features.

Quality is always preferred over quantity.

---

# What Project Phoenix Is

Project Phoenix is:

- Lightweight
- Recovery focused
- Bash based
- Open source
- Privacy respecting

---

# What Project Phoenix Is Not

Project Phoenix is not:

- Enterprise backup software
- Cloud backup software
- A monitoring platform
- A replacement for Docker

It exists for one purpose:

Helping people recover their self-hosted environments quickly and confidently.

---

# Final Principle

Every release should make recovery easier than the previous release.
